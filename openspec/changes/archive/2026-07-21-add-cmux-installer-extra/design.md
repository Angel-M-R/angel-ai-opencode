## Context

The installer models integrations that require behavior beyond catalog copying as hardcoded extras. Every current TUI extra starts selected, `--all` selects every extra, and `PlanInstallation` and `ApplyInstallation` share preparation and reconciliation logic. File reconciliation already creates a backup before replacing divergent content and treats identical content as unchanged.

The cmux integration consists of two OpenCode plugins generated by cmux: `cmux-session.js` for notifications and `cmux-feed.js` for Feed. Angel AI will own pinned copies generated by cmux 0.64.20 rather than invoking `cmux hooks` or allowing cmux to manage the installed files.

## Goals / Non-Goals

**Goals:**

- Expose one cmux extra that is off by default in the interactive installer and installs both plugins as a unit.
- Fail preflight before mutations when selected installation cannot find `cmux` in `PATH`.
- Install pinned vendored files into `plugins/` through the existing plan/apply and reconciliation pipeline.
- Preserve backup behavior and make successful retries no-ops for files already at the desired snapshot.
- Record enough provenance to identify cmux 0.64.20 as the exact snapshot source.

**Non-Goals:**

- Running `cmux hooks`, downloading plugin files, or tracking the latest cmux release at install time.
- Registering plugins in `opencode.json` or changing any other OpenCode configuration.
- Removing installed files when the extra is unselected or adding an uninstall workflow.
- Testing cmux plugin internals, cmux itself, or end-to-end notification and Feed behavior.

## Decisions

### Model the default explicitly on each extra

Extend the extra descriptor with a default-selection property and initialize the TUI from it. Existing extras retain their current selected default; the new cmux extra sets it to false. Explicit select-all actions, including the existing `--all` mode, continue to mean all extras.

This is preferred over a cmux-specific exception in the TUI because descriptor-owned defaults keep selection policy next to each option and remain extensible.

### Vendor one immutable cmux 0.64.20 snapshot pair

Store `cmux-session.js` and `cmux-feed.js` under a dedicated cmux integration asset directory together with English provenance that identifies the generator and exact version. Installation reads only these packaged assets; it neither executes `cmux hooks` nor regenerates them.

This is preferred over runtime generation or download because Angel AI remains the source of truth and installs are reproducible. Updating to a later cmux snapshot requires an intentional repository change.

### Treat cmux availability as a selected-extra preflight

Add a prerequisite check using an injectable executable lookup and run it for both planning and application when, and only when, the cmux extra is selected. Application performs this check with the other preflight checks before any package installation or file write. Failure identifies the selected cmux extra and explains that `cmux` must be available in `PATH`.

This is preferred over discovering the problem after files are copied because preflight must block an unusable selected integration without partial mutations.

### Prepare direct plugin files without configuration registration

When selected, prepare both vendored files with targets `<ConfigDir>/plugins/cmux-session.js` and `<ConfigDir>/plugins/cmux-feed.js`. Append them to the same prepared-file list used by other installer assets. Do not add plugin entries or any other patch to `opencode.json`; OpenCode's plugin-directory auto-discovery is the sole registration mechanism.

Using the common prepared-file pipeline is preferred over cmux-specific writes because planning, atomic replacement, backups, reporting, and unchanged detection remain consistent.

### Reuse file reconciliation for drift and retries

Existing missing targets are created, divergent targets are backed up before replacement, and byte-identical targets are reported unchanged without another backup. If application stops after writing only one plugin, a retry reconciles each target independently: the completed file remains unchanged and the missing or divergent file is repaired.

No deselection cleanup is added. An absent or false cmux selection contributes no prepared files, so previously installed files remain untouched.

## Risks / Trade-offs

- [Vendored plugins can lag cmux releases] → Pin and document cmux 0.64.20; require deliberate snapshot updates.
- [Plugin compatibility may change outside the installer] → Keep validation limited to the declared `cmux` executable prerequisite and preserve exact known assets.
- [A two-file apply can fail between writes] → Reconcile independently and rely on idempotent retries to converge without rewriting the completed file.
- [Adding per-extra defaults changes shared selection code] → Preserve true defaults for every existing extra and add focused selection regression coverage.

## Migration Plan

1. Add the cmux extra metadata and preserve defaults for existing extras.
2. Add the pinned plugin assets and provenance record.
3. Add selected-extra preflight and prepare both plugin targets through the shared pipeline.
4. Add focused installer tests for selection, plan/apply, preflight, backup creation, and idempotent retry.

Rollback consists of reverting the installer change. Files already installed on user machines remain in place by design and can be removed manually if desired.
